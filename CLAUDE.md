## Project Overview

NightScope is a macOS 15.0 (Sequoia) / iOS 18.0+ stargazing app that aggregates location, weather, light pollution, and astronomical data to help users find the best stargazing conditions.

- macOS target: `NightScope/`
- iOS (iPhone) target: `NightScopeiOS/`
- Both managed in a single Xcode project (`NightScope.xcodeproj`)
- Language: Swift 6 (strict concurrency)

## Build & Test Commands

```bash
# macOS build
xcodebuild -quiet -project NightScope.xcodeproj -scheme NightScope \
  -destination 'platform=macOS' build

# macOS tests (all)
xcodebuild -quiet -project NightScope.xcodeproj -scheme NightScope \
  -destination 'platform=macOS' CODE_SIGNING_ALLOWED=NO CODE_SIGNING_REQUIRED=NO test

# iOS Simulator build
xcodebuild -quiet -project NightScope.xcodeproj -target NightScopeiOS \
  -sdk iphonesimulator -configuration Debug \
  CODE_SIGNING_ALLOWED=NO CODE_SIGNING_REQUIRED=NO build
```

Single test class: append `-only-testing NightScopeTests/<TestClassName>` to the test command.

## Architecture

### Core Data Flow

`AppController` is the central state coordinator. It owns all services and publishes `ObservationState` snapshots to the view layer.

```
AppController
├── LocationController       (GPS, MapKit search, reverse geocoding)
├── WeatherKitService        (Apple WeatherKit, protocol: WeatherProviding)
├── LightPollutionService    (reads bortle_map.bin, protocol: LightPollutionProviding)
├── NightCalculationService  (wraps MilkyWayCalculator async, protocol: NightCalculating)
└── FavoriteLocationStore    (UserDefaults / iCloud KVStore)
```

`AppRootDependencies` assembles all ViewModels from `AppController` and injects them at app startup via `AppRootStore`.

### Calculation Pipeline

`MilkyWayCalculator` — pure, synchronous, no I/O — computes galactic center position, sun/moon altitude, and viewing windows per night. `NightCalculationService` wraps it in `Task(priority:)` for async use by `AppController`.

`StarGazingIndex` is derived after both async calculations (night summary) and external data (weather + Bortle class) are available. Recomputed in `recomputeStarGazingIndex()` / `recomputeUpcomingIndexes()` whenever any input changes.

### Protocol Abstractions

Key services are protocol-backed for testability:
- `WeatherProviding` — weather fetch + `weatherByDate` publisher
- `NightCalculating` — `calculateNightSummary` / `calculateUpcomingNights`
- `LightPollutionProviding` — `bortleClass` publisher
- `LocationProviding` — full location state interface used by ViewModels
- `FavoriteLocationStoring` — CRUD for saved locations

Test doubles (`MockLocationController`, `MockLightPollutionService`, etc.) live in `NightScopeTests/ViewModelTestSupport.swift`.

### StarMap ViewModel Split

`StarMapViewModel` delegates to three focused files:
- `StarMapComputation.swift` — sky position math
- `StarMapDateLogic.swift` — date/time handling
- `StarMapPresentation.swift` — display state

### iOS Target

`NightScopeiOS/` contains only iOS-specific views (`iOSRootView`, `iOSTodayView`, `iOSForecastView`, `iOSStarMapView`, etc.) and the app entry point. It reuses all models, controllers, and ViewModels from `NightScope/`.

## Bundled Data Files

Binary/JSON assets in `NightScope/Data/` are generated by Python scripts in `Tools/`. Do not edit them by hand.

| File | Generator | Source |
|---|---|---|
| `bortle_map.bin` | `Tools/generate_bortle_map.py` | Falchi et al. 2016 (CC BY 4.0) |
| `elevation_global.bin.z` | `Tools/prepare_srtm.py` | Copernicus DEM (CC BY 4.0) |
| `elevation_japan.bin.z` | `Tools/prepare_srtm.py --region japan` | Copernicus DEM (CC BY 4.0) |
| `stars_fill.json` | `Tools/generate_stars.py` | Yale BSC5 (Public Domain) |
| `constellations_iau.json` | `Tools/generate_constellations.py` | d3-celestial BSD-3 commit `7e720a3` |

If elevation binaries are absent, `TerrainService` returns nil and the app runs without horizon obstruction calculations.

## Key Conventions

- All controllers and ViewModels are `@MainActor`. Services that do async I/O use `Task` internally and publish results back on the main actor.
- `ObservationState` is a value-type snapshot published from `AppController` to avoid multiple `@Published` subscriptions in views.
- Batch state mutations use `performObservationStateBatchUpdate {}` to emit one `ObservationState` change instead of many.
- `DashboardCommandBridge` is a Combine-based bridge for the macOS Dashboard window to push selections into the main window.
- Location changes cancel in-flight calculation tasks and restart them atomically via `scheduleLocationChangeHandling()`.
- iCloud sync for favorites is opt-in (`iCloudSyncEnabled` UserDefaults key); one-time migration from local store runs on first enable.

## Development Notes

- Before changes likely to exceed 200 lines, plan first and propose to the user.
- WeatherKit requires an active Apple Developer Program membership; tests must pass `CODE_SIGNING_ALLOWED=NO`.
- Attribution for CC BY 4.0 data sources is displayed in Settings → Data Sources & Credits and via `WeatherAttributionBadge`.
- Verify API license terms before adding or modifying external data sources.
